# Non-Sponsored Calls (Sync Fee Calls)

Non-Sponsored calls, also known as Sync Fee Calls, are the simplest way to pay for relay services. However, they delegate all security (reentrancy/replay protection etc.) and payment logic to the target smart contract. You can use ERC-2771 to achieve out-of-the-box security and authentication. Relay costs are covered in either native or ERC-20 tokens and they are paid synchronously during the relay call.

## Implementation Steps

### 1. Deploy a GelatoRelayContext compatible contract

Import GelatoRelayContext in your target contract to inherit callWithSyncFee functionalities:

```solidity
import {
    GelatoRelayContext
} from "@gelatonetwork/relay-context/contracts/GelatoRelayContext.sol";

contract TargetContract is GelatoRelayContext {
    function example() external onlyGelatoRelay {
        // _yourAuthenticationLogic()

        // Payment to Gelato
        // NOTE: be very careful here!
        // if you do not use the onlyGelatoRelay modifier,
        // anyone could encode themselves as the fee collector
        // in the low-level data and drain tokens from this contract.
        _transferRelayFee();
    }

    function exampleFeeCapped(uint256 maxFee) external onlyGelatoRelay {
        // Remember to autheticate your call since you are not using ERC-2771
        // _yourAuthenticationLogic()

        // Payment to Gelato
        // NOTE: be very careful here!
        // if you do not use the onlyGelatoRelay modifier,
        // anyone could encode themselves as the fee collector
        // in the low-level data and drain tokens from this contract.
        _transferRelayFeeCapped(maxFee);
    }
}
```

### For ERC2771 Sync Fee Calls (Recommended)

Import GelatoRelayContextERC2771 in your target contract to inherit ERC2771 functionalities with callWithSyncFee:

```solidity
import {
    GelatoRelayContextERC2771
} from "@gelatonetwork/relay-context/contracts/GelatoRelayContextERC2771.sol";

contract TargetContractRelayContextERC2771 is GelatoRelayContextERC2771 {
    mapping (address=>bool) public caller;

    function increment() external onlyGelatoRelayERC2771 {
        // Payment to Gelato
        // NOTE: be very careful here!
        // if you do not use the onlyGelatoRelayERC2771 modifier,
        // anyone could encode themselves as the fee collector
        // in the low-level data and drain tokens from this contract.
        _transferRelayFee();
        // _getMsgSender() will fetch the original user who signed the relay request.
        caller[_getMsgSender()] = true;
    }
    
    function incrementFeeCapped(uint256 maxFee) external onlyGelatoRelayERC2771 {
        // Payment to Gelato
        // NOTE: be very careful here!
        // if you do not use the onlyGelatoRelayERC2771 modifier,
        // anyone could encode themselves as the fee collector
        // in the low-level data and drain tokens from this contract.
        _transferRelayFeeCapped(maxFee);
        // _getMsgSender() will fetch the original user who signed the relay request.
        caller[_getMsgSender()] = true;
    }
}
```

## Using ERC-20 Permit for Gasless Payments

ERC-20 tokens that implement the permit function (EIP-2612) enable gasless transactions by allowing users to authorize token spending through off-chain signatures instead of requiring an on-chain approval transaction. This eliminates the need for users to hold native tokens for gas fees when paying with ERC-20 tokens.

### Understanding ERC-20 Permit

The permit function allows token holders to approve spending limits through cryptographic signatures rather than transactions. This signature-based approval can be submitted alongside your main transaction, enabling single-transaction flows where users can pay fees in ERC-20 tokens without prior approval transactions.

### Checking Permit Support

Before implementing permit functionality, verify that your target ERC-20 token supports the permit function. You can check this by:

```javascript
// Check if token implements permit function
const hasPermit = await token.supportsInterface("0x9d8ff7da"); // EIP-2612 interface ID
// Or try calling the permit function signature
try {
    await token.permit.staticCall(/* test parameters */);
    // Token supports permit
} catch {
    // Token doesn't support permit
}
```

### Contract Implementation with Permit

When using permit in your smart contract, you can combine the permit call with your main function execution:

```solidity
import {
    GelatoRelayContextERC2771
} from "@gelatonetwork/relay-context/contracts/GelatoRelayContextERC2771.sol";
import {IERC20Permit} from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Permit.sol";

contract TargetContractWithPermit is GelatoRelayContextERC2771 {
    function executeWithPermit(
        address token,
        uint256 amount,
        uint256 deadline,
        uint8 v,
        bytes32 r,
        bytes32 s
    ) external onlyGelatoRelayERC2771 {
        address user = _getMsgSender();
        
        // Execute permit to allow contract to spend user's tokens
        IERC20Permit(token).permit(user, address(this), amount, deadline, v, r, s);
        
        // Your business logic here
        // Transfer tokens for your application's purpose
        IERC20(token).transferFrom(user, address(this), amount);
        
        // Pay relay fee
        _transferRelayFee();
    }
}
```

### Frontend Implementation with Permit

Generate the permit signature on the frontend before sending the relay request:

```javascript
import { GelatoRelay, CallWithSyncFeeERC2771Request } from "@gelatonetwork/relay-sdk";

// Helper function to generate permit signature
async function signPermit(signer, token, amount, spender, deadline, chainId) {
    const domain = {
        name: await token.name(),
        version: "1",
        chainId: chainId,
        verifyingContract: token.target
    };
    
    const types = {
        Permit: [
            { name: "owner", type: "address" },
            { name: "spender", type: "address" },
            { name: "value", type: "uint256" },
            { name: "nonce", type: "uint256" },
            { name: "deadline", type: "uint256" }
        ]
    };
    
    const nonce = await token.nonces(signer.address);
    
    const values = {
        owner: signer.address,
        spender: spender,
        value: amount,
        nonce: nonce,
        deadline: deadline
    };
    
    const signature = await signer.signTypedData(domain, types, values);
    return ethers.Signature.from(signature);
}

// Main function to execute with permit
const executeWithPermit = async () => {
    const signer = await provider.getSigner();
    const amount = ethers.parseUnits("100", 6); // 100 USDC
    const deadline = Math.floor(Date.now() / 1000) + 3600; // 1 hour from now
    const { chainId } = await provider.getNetwork();
    
    // Generate permit signature
    const sig = await signPermit(
        signer,
        tokenContract,
        amount,
        targetContractAddress,
        deadline,
        chainId
    );
    
    const { v, r, s } = sig;
    
    // Encode function call with permit parameters
    const data = encodeFunctionData({
        abi: targetAbi,
        functionName: "executeWithPermit",
        args: [tokenAddress, amount, deadline, v, r, s]
    });
    
    // Send relay request
    const request: CallWithSyncFeeERC2771Request = {
        chainId: BigInt(chainId),
        target: targetContractAddress,
        data: data,
        user: signer.address,
        feeToken: tokenAddress,
        isRelayContext: true,
    };
    
    const relayResponse = await relay.callWithSyncFeeERC2771(request, signer);
};
```

### Benefits of Using Permit

Using ERC-20 permit with Gelato Relay provides several advantages:

- Single Transaction Flow: Users can approve and execute in one transaction
- Better User Experience: No separate approval transaction required
- Gas Efficiency: Reduces the total number of transactions needed
- Enhanced Security: Permits can have expiration times and specific amounts

### Important Considerations

When implementing permit functionality, keep these points in mind:

- Not all ERC-20 tokens support permit - always check compatibility first
- Permit signatures have deadlines - ensure adequate time for transaction execution
- Each permit signature can only be used once due to the nonce mechanism
- Always validate permit parameters in your smart contract to prevent replay attacks

This permit-based approach enables seamless user experiences where token approvals and main functionality can be executed in a single gasless transaction through Gelato Relay.
